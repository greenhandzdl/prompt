## 项目认知：建立 AI 对项目的理解

在让 AI 介入现有项目之前，首先需要帮助 AI 建立对项目的整体认知。这一步骤的核心是让 AI 深度分析代码库，提取项目的架构模式、编码风格和关键约定。

### 实践方法

让 AI 系统性地分析项目并生成开发指导文档，该文档将作为后续协作的"项目上下文"。

### 示例提示词

```prompt
现在我们需要为该项目的 AI 协作开发建立起基本认知和指导文档，我需要你帮我深度研究并解析当前项目的所有实现，从中整理出当前项目多开发指导规则文档，尽可能多地审查清楚所有的代码组织架构和项目整体实现风格，确保编写的文档具有可操作的实践意义

你需要注意：
1. **禁止臆测任何内容**：所有文档内容必须有据可依且具有实际意义！
2. 不要在文档中赘述过于基本的框架用法和技术常识，你应当认为你所编写的内容服务于和你类似智能的开发助手，而不是初学者，只需要写最有用的针对该项目的项目开发知识即可
3. 重点关注项目独有的架构设计、业务关系、重要约定、接口默契、公用工具等
4. 始终保持更新: 文档中需要明确要求当发生变更时始终保持文档内容为符合真实情况的最新状态
5. 编写文档结果保存在 .cursor/rules/background.mdc 中
```

### 关键要点

- **知识检验**：确保 AI 能够对项目建立正确的认知，及时修正不正确的理解
- **聚焦特性**：关注项目特有的模式，而非通用知识
- **持续更新**：要求随着项目演进，定期更新项目认知文档

## 需求表达：如何有效传达开发任务

有效的需求表达是 AI 辅助编程成功的关键。核心方法论是：**主动描述 + 被动澄清**。

### 核心原则

1. **提供完整需求**：事无巨细地明确告知任务目标、参考对象、相关资源、具体需求等
2. **强制验证需求**：要求 AI 在执行前进行疑点检查，禁止任何模糊不清的需求理解
3. **明确实现质量**：明确表达要求高质量的真实实现，不接受降级实现/占位实现等

### 实践案例

以下是一个实际需求的表达示例，展示如何通过结构化的提示词减少无效迭代和沟通成本。

**任务背景**：开发一个数据规则配置页面，用于配置产品的数据模板和解析规则

**参考原型**：<http://prototype.example.com/platform/management/rules/data-analysis>

#### 实践完整提示词

```prompt
现在我们需要开发 **数据规则配置** 页面，该页面的主要功能是用来为系统配置产品的数据模板和解析规则，服务于进一步的数据处理需求，你的主要参考目标页面是数据清洗页面 @data-clean 和分类识别页面 @category-identify  ，必须先严格对齐参考页面的组织风格、接口调用、样式细节等！

你可能需要的接口文档有： @数据解析相关接口定义.jsonc  @分类识别相关接口.jsonc @标准字段相关接口定义.txt

具体需求：

1. 数据规则配置页面使用分页查询接口，参考原型图呈现的表格样式使用恰当的控件展示重要字段
2. 原型图仅供参考，注意有以下细节与原型图不匹配，以具体要求为准：
  1. 表格的操作列中，“查看”、“配置”、“更多” 三个操作按钮简化为 “配置” 和 “删除” 两个按钮，点击配置直接弹出该行规则的配置菜单
  2. 表格上方的 “双向测试器” 暂时不做
  3. “配置” 抽屉下方的按钮只有取消和保存，不需要“保存并测试”，并且滚动到底部的执行配置中，“规则状态”只需要 启用/禁用 即可，不需要 “测试中” 的状态
3. 点击 "配置" 后弹出 "编辑数据解析规则" 抽屉，这个抽屉的内容比较复杂，我会逐一解释每一部分的功能和作用，你需要结合原型图理解
  1. 第一部分首先需要用户填写好规则名称和适用分类，你可以依据分类识别相关接口文档中的关键字搜索接口来设计这一控件
  2. 第二部分是分类对应的关联规则参考，在适用分类发生变化时，需要调用分类识别接口查询到对应的关联规则，按照原型图展示的呈现供用户参考，"智能同步"暂时不可点击，先不做功能，让用户手动配置
  3. 第三部分是最重要的**结构段定义**：每一段可以配置前置分隔符、段含义、段标识、段标识描述、是否显示段标识；其中段含义的枚举会影响其跟随的选项，如果是关联字段时，下方出现"关联字段"选项，从 @标准字段相关接口定义.txt 中获取合适的接口查询供用户选择，如果是固定字符串，下方出现"固定值"文本输入框，让用户自行输入
  4. 第四部分是模板预览和执行配置，

你的实现起点是 @index.vue

务必先严格审查所有文档和依赖资源实现，先全面审查验证好所有实现细节，如有疑问或者需求描述不清晰的地方务必直接提出，不得隐瞒问题或捏造虚假实现/占位实现，准备好开始执行任务
```

## 知识传承：追踪迭代中的经验积累

每个功能的开发过程都包含宝贵的决策、踩坑和解决方案。及时将这些经验文档化，可以为后续的功能迭代和问题排查提供重要参考。

### 实践方法

在完成一个功能模块后，让 AI 整理开发过程中的关键信息，形成"功能演进文档"。

### 示例提示词

```prompt
现在你需要详细整理我们前面的对话内容和该页面的具体实现情况，
新建一份文档保存到 @数据规则配置功能演进文档.md 中，用于未来功能变更和追溯使用。

你需要完整、全面地记录：
- 需求描述：功能的目标和背景
- 参考情况：功能实现的参考目标
- 依赖情况：使用的接口、组件、工具库
- 开发情况：核心实现思路和关键代码位置
- 注意点：特殊处理逻辑和业务规则
- 遇到的问题：开发过程中的障碍和难点
- 解决方案：具体的解决思路和方法
- 沉淀经验：可复用的模式和避坑指南

注意：不要记流水账，不要直接抄写具体代码实现。
```

### 关键要点

- **及时记录**：在功能完成后立即整理，留存上下文重要信息
- **持续维护**：当功能发生重大变更时，同步更新演进文档

## 开发技巧：提升 AI 协作效率

### 1. 应对机器幻觉

**问题表现**

AI 模型在训练时学习了大量不同版本的代码，容易产生混淆，典型表现包括：

- 混淆框架不同版本的 API（如 Vue2 vs Vue3）
- 混淆依赖库的版本差异（如 Python 库的不同版本）
- 对冷门工具的理解不准确
- 根据命名规律臆测"可能存在"但实际不存在的函数
- 传递不合法的参数或使用已废弃的 API
- 业务常识理解出现错误

**解决方案**

1. **强化项目认知**

   - 在项目初期建立详细的技术栈文档
   - 明确框架版本、依赖版本等关键信息
   - 明确项目的实际环境约束和业务背景

2. **引导实证检索**

   - 要求 AI 放弃对已有知识的信任，直接联网搜索
   - 查阅官方文档和社区解决方案
   - 验证 API 的当前版本是否支持某个特性

3. **提供源码参考**

   - 必要时直接引用依赖的实现源码
   - 让 AI 基于真实实现进行分析和诊断
   - 避免基于记忆的臆测

4. **强制类型约束**

   - 优先使用 TypeScript 而非 JavaScript
   - 对 Python 项目强制要求类型注解（使用 Pylance/Pyright/Ruff 检查）
   - 要求 AI 对所有函数、方法和变量进行完整的类型标注和禁止使用 Any
   - 利用类型系统在编码阶段发现潜在问题，避免运行时错误

### 2. 合理使用回溯历史节点

在 Cursor IDE 中，我们可以通过回溯到历史任一节点来快速恢复之前的项目状态，这可以让我们非常安全地使用 AI 生成的内容进行测试评估，而不用担心丢失之前的工作内容，常见的应用场景有:

**在生成内容完全不可用或对项目造成过度破坏时恢复:**

在 AI 编码过程中，可能因为需求表述不清晰、AI 生成幻觉等原因，导致生成的内容完全不可用或对项目造成难以预测的影响，此时我们可能通过回溯到之前的某个历史节点来安全地恢复之前的项目状态

**回到上下文质量更高的时刻调整策略:**

当我们刚为需求编写完完整详细的提示词文档时，此时上下文“负担”最小，但我们描述的需求不一定能够完整覆盖实现的所有细节，此时可以直接直接进行提交开发，如果在开发过程中发现 AI 掉进坑里，或者对关键需求理解有偏差，我们可以直接进行记录然后回溯到最开始时刻进行调整，相当于预先把 AI 会遇到的坑都告知他了，在更小的上下文压力下通常会有更好的发挥

### 3. 用户提示词

用户提示词存储于 Cursor 应用本身，对所有项目均生效，对 AI 回复偏好风格影响最为明显，高质量的用户提示词往往能显著提高 AI 的生成质量，大幅降低无效对话和返工成本

具体提示词参考查阅 `recommend-user-rule.md` 文件

### 4. 自动接口文档检索 MCP

Cursor 支持通过 MCP 调用所有可用的 MCP 服务，结合 ApiFox 的文档 MCP 能力，可以实现自动查阅接口文档的功能: [Apifox MCP 文档](https://docs.apifox.com/6327888m0)

### 5. TypeScript 扩展自动 Linter 检查范围和自动修复格式

在前端项目中，出于性能优化考虑，Cursor (vscode 系 IDE) 不会自动对当前项目中的所有 TypeScript 代码进行类型检查，而只会对用户真实打开编辑的文件进行检查，这通常会导致 AI 在编辑新文件时产生大量的 Linter 错误，并且 AI 不会自动读取这些错误进行修复，以下配置可以扩展自动 Linter 检查范围，让未打开的文件中的错误也能及时被 AI 读取到并修复

```jsonc
// 在项目目录下的 .vscode/settings.json 中添加以下配置
{
  "eslint.format.enable": true,
  "editor.formatOnSave": true,
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "always"
  },
  "typescript.tsserver.experimental.enableProjectDiagnostics": true
}
```

## 设计原则：面向 AI 开发的未来工程实践

### 1. 项目语义化设计

AI IDE（如 Cursor）通常使用 RAG（检索增强生成）技术来查找相关代码。良好的语义化设计可以显著提升 AI 的代码理解和检索效率。

#### 1.1 功能导向型的命名架构

**文件和目录命名**

传统软件项目常以“逻辑分层结构”划分代码目录，例如：

```
project-root/
└── src/
    ├── core/
    │   ├── config.py
    │   ├── logger.py
    │   └── database.py
    ├── models/
    │   ├── user.py
    │   └── order.py
    ├── services/
    │   ├── user.py
    │   └── order.py
    ├── routers/
    │   ├── user.py
    │   └── order.py
    ├── schemas/
    │   ├── user.py
    │   └── order.py
    ├── systems/
    │   └── other.py
    └── tools/
        └── common.py
```

这种按层次划分虽容易辨识结构，但在使用 AI 辅助和 RAG 检索资源时，查找关联代码会效率较低。为提升 AI 辅助开发协作效率，建议采用**功能导向型**架构，将相关代码集中到具体功能目录下，结构如下：

```
project-root/
└── src/
    ├── core/
    │   ├── config.py
    │   ├── logger.py
    │   └── database.py
    └── features/
        ├── user/
        │   ├── router.py
        │   ├── service.py
        │   ├── store.py
        │   ├── types.py
        │   └── utils.py
        └── order/
            ├── router.py
            ├── service.py
            ├── store.py
            ├── types.py
            └── utils.py
```

通过以功能为中心组织代码，可以让 AI 更容易定位和检索与某项业务相关的所有实现、类型、存储、工具等文件，甚至可以把功能相关的设计提示词、迭代追踪记录直接保存到该功能模块下，从而更高效地支持开发、维护和知识传递。

**函数和变量命名**

```typescript
// ✅ 好的命名：语义清晰，易于搜索和理解
async function fetchOrdersByDateRange(
  startDate: Date,
  endDate: Date
): Promise<Order[]> {
  // ...
}

const unpaidOrderCount = orders.filter(
  (order) => order.status === "unpaid"
).length;

// ❌ 糟糕的命名：缺乏语义，难以理解和检索
async function getData(d1: Date, d2: Date): Promise<any[]> {
  // ...
}

const count = list.filter((item) => item.status === "unpaid").length;
```

#### 1.2 编写自解释的代码

**使用类型系统传达意图**

```typescript
// ✅ 通过字符串枚举替代数值枚举来清晰传达意图
type OrderStatus = "pending" | "paid" | "shipped" | "completed" | "cancelled";

interface Order {
  id: string;
  userId: string;
  status: OrderStatus;
  totalAmount: number; // 单位：分
  createdAt: Date;
}

function canRefund(order: Order): boolean {
  return order.status === "paid" || order.status === "shipped";
}

// ❌ 缺乏语义化枚举表达，意图不明确
function canRefund(order: any): boolean {
  return order.status === 1|| order.status === 2;
}
```

**适度的注释**

注释应该解释"为什么"而非"是什么"，同时考虑 RAG 检索需求：

```typescript
/**
 * 计算订单实付金额
 *
 * 业务规则：实付金额 = 商品总额 - 优惠券折扣 - 积分抵扣 + 运费
 * 注意：优惠券和积分不能同时使用在运费上
 *
 * @关键词 订单金额计算 优惠券 积分抵扣
 */
function calculateActualAmount(order: Order): number {
  // 实现...
}
```

注意注释中的 `@关键词` 标记，这可以帮助 RAG 系统更准确地检索到相关代码。

#### 1.4 服务于 RAG 检索的代码组织

**README 和模块说明**

在每个重要模块下添加 README.md，说明模块的职责和关键概念：

```markdown
# 订单管理模块

## 核心功能

- 订单列表查询和筛选
- 订单详情查看
- 订单状态流转
- 订单退款处理

## 关键文件

- `services/orderApi.ts`: 订单相关 API 调用
- `components/OrderList.vue`: 订单列表组件
- `store/orderStore.ts`: 订单状态管理

## 业务规则

- 订单状态流转：待支付 → 已支付 → 已发货 → 已完成
- 退款仅支持已支付和已发货状态
```

### 2. 提示词资产管理

高质量的提示词是 AI 协作开发中最重要的资产。通过系统化管理提示词，可以积累最佳实践，提升团队协作效率。推荐的提示词维护目录结构如下:

```
.cursor/
├── rules/                        # 开发规则和约定
│   ├── background.mdc           # 项目背景和技术栈
│   ├── coding-standards.mdc     # 编码规范
│   ├── architecture.mdc         # 架构设计原则
│   └── modules/                 # 模块级规则
│       ├── auth.mdc            # 认证模块规则
│       └── payment.mdc         # 支付模块规则
│
├── trace/                        # 功能追踪文档
│   ├── overview.md              # 项目整体演进历史
│   └── features/                # 功能级追踪
│       ├── user-management.md
│       └── order-system.md
│
└── module // 如果使用功能导向组织项目，可以直接放在对应的功能模块中
```

**目录说明**

- **rules/**：存放开发规则，AI IDE 会自动读取这些规则作为上下文
- **trace/**：功能演进追踪，记录决策和经验
- **module/**：模块级文档，详细说明各模块的实现 (如果使用功能导向组织项目，可以直接放在对应的功能模块中)

### 3. 前后端一体化架构

在传统的软件开发中，我们往往会将前后端代码分别放在不同的仓库中，这可以方便地进行前后端分工分离开发，但随之而来的代价高昂的沟通成本，但在 AI 辅助编程时代这一传统观念可能不再适用，未来的 AI 深度集成项目中，我们建议采用前后端一体化的全栈架构，配合端到端注解、RPC调用等方式来实现复杂的前后端协同自动开发能力
